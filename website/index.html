What is DFIR - and what problem does COUNSEL solve?

When an organization is breached, someone has to piece together the full story from hundreds of digital artifacts: which programs auto-started on boot, which files ran and when, what the machine was talking to over the network, what was sitting in memory. A skilled DFIR analyst can do this - but it takes days. Automating it with AI is the goal of this hackathon. The problem is that AI agents tend to assert confident findings based on a single ambiguous log line. COUNSEL's answer to this is architectural: a finding cannot reach "confirmed" unless two independent sources agree, enforced by a deterministic engine that the AI cannot override.
Without COUNSEL
  • x AI sees one piece of evidence and asserts a confident finding
  • x No way to trace an assertion back to raw evidence
  • x Agent can run shell commands, write files, be manipulated
  • x Uncertainty is hidden inside a confident-sounding sentence
  • x Runs are not reproducible or independently verifiable
With COUNSEL
  • v Finding only confirmed when two independent sources agree
  • v Every sentence traces to an exact tool call and raw output hash
  • v Agent has no shell, no write access, no dangerous primitives at all
  • v Uncertainty is explicit: state shows INFERENCE or UNRESOLVED
  • v Any finding can be replayed and hash-verified by a third party

How It Works

A bounded investigation loop - not a chat session

COUNSEL is not a chatbot that happens to call forensic tools. It runs a structured investigation cycle with defined phases, a maximum iteration cap, and graceful degradation when the budget runs out.

01

TRIAGE

The agent scans the evidence directory and runs initial forensic tools to build a first picture. Registry hives, prefetch records, and event logs are checked. Everything suspicious becomes a candidate claim in OBSERVED state.

02

PROPOSE

Based on triage, the corroboration engine assigns states. A suspicious registry key becomes OBSERVED. When a prefetch record matches, it becomes INFERENCE. The engine controls state - not the AI.

03

VERIFY

The agent runs additional tools to corroborate or contradict each INFERENCE. It specifically seeks sources independent of what it has already seen - the same reasoning a good analyst uses.

04

GATHER + Self-Correction

Gap detection identifies unchecked high-weight signals. When new evidence contradicts an earlier ruling, a RULING CHANGE is announced in the terminal. Genuine self-correction, on camera, driven by the engine.

05

VERDICT + SIGN

The agent writes its final verdict. After it exits, the launcher - which holds the signing key in a separate process - applies an Ed25519 signature to the full audit ledger. The agent never handles the key.

The 5-State Claim Model

Every finding has exactly one state. The engine decides - not the AI.

States only advance when evidence meets mathematical thresholds. The AI reads state, never writes it.

Why a state machine matters: Without one, an AI agent can say "the attacker definitely did X" based on a single ambiguous log line. With the 5-state model, that same line produces OBSERVED - not a verdict. To reach CORROBORATED, the engine requires a support score of at least 0.80 from at least two independent source groups.
OBSERVED

Single Artifact

One source noted it. Not enough to assert. Investigation continues.

INFERENCE

Some Support

Evidence points this way but support is below 0.80, or fewer than two independent groups have contributed.

CORROBORATED

Confirmed

Two or more independent sources agree. Support is at least 0.80. The only state safe to report as a definitive finding.

CONTRADICTED

Conflicts

A source actively contradicts this claim. Contradiction score is at least 0.60. Do not report. Investigate the conflict.

UNRESOLVED

Inconclusive

Investigation budget ran out or evidence was unavailable. Honest about the limit. Not reported as a positive finding.

The math behind it: Support score = 1 - (1 - w1) x (1 - w2) x ... across independent groups. A registry signal with weight 0.95 and a prefetch signal with weight 0.85 produce support = 1 - (1 - 0.95)(1 - 0.85) = 1 - 0.0075 = 0.9925. Above 0.80 and from two independent groups: CORROBORATED. This is called Noisy-OR.
The Corroboration Rule DSL

Write DFIR knowledge in plain YAML.

Any DFIR analyst can add a new corroboration rule. Rules are YAML files in counsel/rules/. Bad rules are rejected at startup with a clear error - they never load silently.

What this rule says: To confirm that malware persisted via a Windows run key, the registry parser AND the prefetch reader must independently agree. The registry alone is not enough. Prefetch alone is not enough. Together they reach CORROBORATED. If the file is missing from disk, the contradictions block fires and the claim becomes CONTRADICTED instead.
rule - Unique ID. Stored in every ledger entry that this rule evaluated.
emits - Which claim types this rule can evaluate. One rule can cover multiple claim types.
signals - Which forensic tools contribute evidence and how strongly (0.0 to 1.0).
independent_of - Groups signals. Signals in different groups are treated as independent sources. Only one signal per group can count.
requires - Optional predicate. The signal only fires if this field condition passes on the tool result record.
contradictions - Sources that actively disprove the claim if their predicate passes.
provenance - Required. Cite your DFIR source. Rules without provenance fail at load time. No citation, no rule.
counsel/rules/persistence_run_key.yaml
rule: persistence_via_run_key
emits:
  - persistence_configured
  - payload_present

signals:
  - artifact: registry.run_keys
    supports: persistence_configured
    weight: 0.95
    independent_of: registry_sources
      # run key present = strong registry signal

  - artifact: prefetch.run_record
    supports: persistence_configured
    weight: 0.85
    independent_of: execution_sources
    requires: "run_count > 1"
      # different group = counts as independent

  - artifact: fs.stat_hash
    supports: payload_present
    weight: 0.90
    independent_of: filesystem_sources
    requires: "exists == true"

contradictions:
  - artifact: fs.stat_hash
    weight: 0.90
    requires: "exists == false"
      # file missing = contradiction fires

provenance: "SANS FOR508 - Registry Persistence;
  Red Canary Threat Detection Report 2023"
Architecture

Three trust boundaries. No exceptions.

Security architecture only matters if it holds when things go wrong. These three boundaries ensure that even a compromised AI model cannot write files, run commands, or access the signing key.

Trusted Launcher

Holds the signing key. Mounts evidence read-only. Spawns the MCP server and agent. Signs the ledger manifest after the agent exits.

B2: key isolation
→

MCP Server

10 typed forensic tools only. Every result is parsed into structured records before the agent sees it. Read-only file system access. Every call logged to the hash-chained ledger.

B1: typed interface only
→

Agent Runtime

Claude Opus 4.8. Can only call the 10 MCP tools. No shell, no exec, no write, no direct evidence access. Cannot touch the signing key.

B3: capability isolation
Why this beats prompt guardrails: A prompt saying "do not run shell commands" can be overridden by a sufficiently clever adversarial input planted inside the evidence files. Removing the shell primitive entirely means there is nothing to override - no instruction can grant a capability that does not exist in the tool list.
RT1

Filename Injection

A file is named to inject instructions. Parse-before-return strips control chars before the AI sees the filename.

RT2

Content Injection

A file contains adversarial instructions. COUNSEL passes only parsed, structured records to the agent. Raw file content is never included in tool results.

RT3-7

Shell, Ledger, Exfil, Replay

Shell escape attempts fail (no shell tool exists). Ledger tampering is detected by hash chain verification. Exfiltration attempts fail (no outbound network tool). Replay attacks produce hash mismatches.

Deploy

One command on SIFT Workstation.

Designed for the SANS SIFT Workstation environment. All required forensic tools are already installed on SIFT. Nothing to configure manually.

SIFT Workstation - Quick Start 
# 1. Clone and install
git clone https://github.com/usv240/counsel
cd counsel
pip install -e .

# 2. Set your Anthropic API key
export ANTHROPIC_API_KEY=sk-ant-...

# 3. Mount evidence read-only  (never modify original evidence)
sudo mount -o ro,loop /path/to/evidence.E01 /mnt/evidence

# 4. Generate signing key pair  (one-time)
counsel keygen ~/.counsel/keys

# 5. Run the investigation
counsel investigate /mnt/evidence \
  --signing-key ~/.counsel/keys/counsel_signing.pem \
  --output-dir ./results

# 6. Replay any finding and verify its hash  (replace 7 with any seq number)
counsel replay results/a3f7e2b9/counsel-ledger.jsonl 7

# 7. Run the full red-team test suite
counsel redteam /mnt/evidence

# 8. Enable analyst training mode
counsel investigate /mnt/evidence --training-mode
